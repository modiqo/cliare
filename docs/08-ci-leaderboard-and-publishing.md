# 08 - CI, Leaderboard, and Publishing

> **Scope:** How CLIARE runs in CI, how scorecards are published, and how verified leaderboard entries are produced.
> **Status:** Reference Design

---

## Core Decision

CLIARE runs the target binary in the user's own CI by default.

modiqo does not need to execute arbitrary user binaries for the primary workflow.

Default workflow:

```text
Project CI builds CLI
Project CI runs cliare
Project CI stores artifacts
Project CI optionally publishes scorecard
modiqo leaderboard displays verified scorecard
```

This model is more secure, cheaper, more reproducible, and easier to adopt.

---

## GitHub Action

Example:

```yaml
name: CLIARE

on:
  pull_request:
  push:
    branches: [main]
    tags: ["v*"]

jobs:
  cliare:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
      checks: write
      security-events: write

    steps:
      - uses: actions/checkout@v4

      - name: Build CLI
        run: make build

      - name: Run CLIARE
        uses: modiqo/cliare-action@v1
        with:
          binary: ./dist/mycli
          mode: guard
          baseline: .cliare/baseline.scorecard.json
          publish: ${{ startsWith(github.ref, 'refs/tags/') }}
```

Action outputs:

- `.cliare/evidence.jsonl`
- `.cliare/shape.json`
- `.cliare/scorecard.json`
- `.cliare/report.md`
- `.cliare/sarif.json`
- GitHub Step Summary
- PR annotations

---

## CI Modes

### `measure`

Never fails CI. Produces artifacts.

Use for first adoption:

```yaml
mode: measure
```

### `guard`

Fails only on configured regressions.

Use for PRs:

```yaml
mode: guard
baseline: .cliare/baseline.scorecard.json
```

### `certify`

Strict reproducible profile for public scorecard.

Use for releases:

```yaml
mode: certify
publish: true
```

---

## CI Artifacts

Artifacts should be small enough by default.

Store:

- scorecard JSON
- report Markdown
- SARIF
- shape catalog
- redacted evidence log

Large raw stdout/stderr artifacts should be optional.

Retention:

- PR runs: 14 days
- main branch: 90 days
- release tags: permanent if published

## Scorecard Publishing

Publishing command:

```sh
cliare publish .cliare/scorecard.json
```

Optional:

```sh
cliare publish .cliare/scorecard.json --evidence .cliare/evidence.jsonl
```

Scorecard payload:

```json
{
  "project": "owner/repo",
  "cli": "mycli",
  "version": "1.2.3",
  "score": 84,
  "subscores": {},
  "verification": {},
  "artifact_hashes": {},
  "source": {
    "provider": "github_actions",
    "run_id": "...",
    "commit_sha": "...",
    "tag": "v1.2.3"
  }
}
```

The publishing endpoint should accept scorecards, verify schema, validate signatures/attestation if present, and store immutable records.

---

## Attestation

Use CI provider identity where possible.

For GitHub Actions:

- OIDC token
- repository
- commit SHA
- workflow ref
- run id
- tag

Verification should assert:

- scorecard came from claimed repo
- commit/tag matches
- action used a known CLIARE version
- artifact hash matches uploaded content

Do not require attestation for all submissions. Instead, show verification level.


---

## Anti-Gaming Controls

Leaderboard must disclose:

- score model version
- probe profile
- verification level
- evidence availability
- run date
- target version

Red flags:

- scorecard without evidence hash
- unusually narrow probe coverage
- target binary unavailable
- profile not certified
- missing provenance
- score generated by old model

Leaderboard should allow filters:

```text
Show only CI-attested or better
Show only certified profile
Show only score model v1
```

## Hosted Product Path

The OSS project should be complete locally.

Hosted modiqo value can include:

- scorecard hosting
- leaderboard
- historical trends
- team dashboards
- policy management
- enterprise verification
- cross-repo comparison
- Slack/GitHub notifications
- private scoreboards
- deeper benchmark reports

The hosted product should not be necessary for initial adoption.

## CI Failure Output

Failure should be concise:

```text
CLIARE guard failed: score regression -6.4 exceeds threshold -3.0

Primary causes:
- Safety -4.1: new command `env delete` has destructive risk and no dry-run evidence.
- Output -1.8: `project list --json` no longer emits valid JSON.
- Recovery -0.5: unknown flag errors lost suggestions.

Report: .cliare/report.md
```

The PR report can include long form details.

---

## Data Retention and Privacy

Hosted scorecards should not require raw output.

Submission modes:

| Mode | Uploaded |
|------|----------|
| scorecard only | scorecard JSON |
| evidence hash | scorecard plus evidence hash |
| redacted evidence | redacted evidence log |
| full evidence | full evidence, private or certified |

Default public publishing should be scorecard plus artifact hashes.

---

## Initial CI Scope

The initial CI integration should ship:

- GitHub Action
- `measure`, `guard`, `certify`
- Markdown report
- JSON scorecard
- baseline diff
- no hosted dependency

Hosted leaderboard can follow once the score model is stable enough.
