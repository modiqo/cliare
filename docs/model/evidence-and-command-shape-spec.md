# 05 - Evidence And Command Shape Spec

> **Scope:** Current durable artifacts, evidence schema, command-shape schema, command-index schema, artifact map, and context-suite contracts.
> **Status:** Current Implementation

---

## Summary

CLIARE measurement output is file based. The core technical artifacts are:

1. **Evidence log**: `evidence.jsonl`, the runtime record of scheduled probes and completed target processes.
2. **Command shape**: `shape.json`, the inferred command/flag/output/precondition model derived from evidence.
3. **Command index**: `command-index.json` and `command-index.md`, the command-centric projection intended for agents, maintainers, and harnesses.
4. **Score and reports**: `scorecard.json`, `report.md`, CI artifacts, persona packets, and issue ledgers derived from the same evidence and shape.
5. **Artifact map**: `artifact-map.json` and `artifact-map.md`, generated on demand by `cliare describe --write`.
6. **Context suite**: `context-suite.json` and `context-compare.md`, generated by `cliare context compare --write`.

The evidence log is the lowest-level proof source. `shape.json` is the inferred model. `command-index.json` is the command lookup table an agent harness should open before deciding whether and how to call a CLI command.

## Evidence Log

File:

```text
evidence.jsonl
```

For context-suite measurements, the file lives under the selected context artifact directory:

```text
<suite-root>/contexts/<context>/evidence.jsonl
```

Each line is one JSON object with these top-level fields:

```json
{
  "schema_version": "cliare.evidence.v1",
  "event_id": "e_000001",
  "timestamp": "2026-06-15T21:11:15.968402Z",
  "kind": "run_started",
  "payload": {}
}
```

Current evidence event kinds are:

- `run_started`
- `probe_scheduled`
- `process_completed`
- `run_finished`

There are no separate `process_started`, `stdout_artifact`, `stderr_artifact`, `filesystem_diff`, `network_event`, `process_event`, `classifier_annotation`, or `redaction_event` records in the current implementation. Filesystem changes are embedded in `process_completed.side_effects`. Network and subprocess tracing are not first-class runtime evidence today.

## Process Completed

`process_completed` embeds process status, bounded output captures, duration, and filesystem side effects:

```json
{
  "schema_version": "cliare.evidence.v1",
  "event_id": "e_000010",
  "timestamp": "2026-06-15T21:11:16.100000Z",
  "kind": "process_completed",
  "payload": {
    "probe_id": "p_000001",
    "argv": ["/repo/target/debug/cliare", "--help"],
    "status": {
      "state": "exited",
      "code": 0
    },
    "duration_ms": 43,
    "stdout": {
      "sha256": "...",
      "bytes": 1024,
      "retained_bytes": 1024,
      "truncated": false,
      "text": "Usage: cliare <COMMAND>..."
    },
    "stderr": {
      "sha256": "...",
      "bytes": 0,
      "retained_bytes": 0,
      "truncated": false,
      "text": ""
    },
    "side_effects": {
      "created": 0,
      "modified": 0,
      "deleted": 0,
      "total": 0,
      "changes": []
    }
  }
}
```

Current process status forms are:

```json
{ "state": "exited", "code": 0 }
{ "state": "timed_out" }
{ "state": "spawn_failed", "error": "..." }
```

Output is retained inline up to `--output-limit-bytes` per stream. CLIARE records a stream hash and byte counts; it does not currently spill large output to separate stdout/stderr artifact files.

## Side Effects

Filesystem side-effect changes appear under `process_completed.payload.side_effects`.

Current change fields:

```json
{
  "kind": "created",
  "region": "xdg_cache",
  "path": "probes/p_000001/xdg-cache/mycli/cache.json",
  "size_bytes": 128,
  "sha256": "..."
}
```

Current change kinds are:

- `created`
- `modified`
- `deleted`

Current regions are:

- `home`
- `workdir`
- `xdg_config`
- `xdg_cache`
- `xdg_data`
- `tmp`

When `--context-workdir` points at a real project directory, CLIARE uses metadata snapshots for that workdir and does not record content hashes for those files.

Host execution currently has no registered snapshot regions, so host-mode side effects are not captured as filesystem diffs.

## Run Finished

`run_finished` records the completed probe count:

```json
{
  "schema_version": "cliare.evidence.v1",
  "event_id": "e_000099",
  "timestamp": "2026-06-15T21:11:30Z",
  "kind": "run_finished",
  "payload": {
    "probes_completed": 96
  }
}
```

Traversal budget posture is recorded in `scorecard.json` and progress logs, not in the `run_finished` payload.

## Command Shape

File:

```text
shape.json
```

Top-level shape fields:

```json
{
  "schema_version": "cliare.command-shape.v1",
  "target": {},
  "commands": [],
  "flags": [],
  "output_contracts": [],
  "gaps": [],
  "model": {
    "name": "cliare-generic-claims-v0",
    "source": "generic claim store with layout evidence, runtime confirmation, and diagnostic probes"
  }
}
```

`shape.json` is an inferred artifact. It is not a user-authored schema and it is not a complete source of truth without `evidence.jsonl`.

## Positional Argument

Positionals appear inside command candidates and command-index entries:

```json
{
  "name": "persona",
  "required": true,
  "variadic": false,
  "evidence": ["e_000023:usage syntax"]
}
```

The current schema records name, requiredness, variadic status, and evidence references. It does not emit a semantic type or JSON Schema value schema for positionals.

## Agent Suitability

`agent_suitability` is the harness-facing routing classification for a command. It is derived from runtime state, preconditions, gaps, and output-contract status.

| Suitability | Meaning | Harness treatment |
|---|---|---|
| `ready` | Runtime-confirmed command with no blocking gaps. | Candidate for automatic routing, subject to harness permissions and task policy. |
| `conditional` | Runtime-confirmed or runtime-recognized command with a satisfiable condition or grammar/diagnostic gap. | Expose when the harness can satisfy the condition or policy allows manual review. |
| `needs_fixture` | Advertised contract or behavior needs safe operands, fixture data, or command-local validation before CLIARE can verify it. | Add fixtures or documented safe operands before routing automatically. |
| `blocked` | Opaque, infrastructure-level, or currently unsatisfied runtime condition blocked safe confirmation or use. | Treat as unavailable until the runtime condition is provisioned or the diagnostic becomes explicit enough to classify. |
| `candidate` | Inferred from help/layout evidence but not runtime-confirmed. | Keep out of automatic routing until deeper measurement or explicit metadata confirms it. |

```json
  "argv_fragment": ["--format", "json"],
  "scope": "command_flag",
  "status": "precondition_blocked",

  "evidence": ["e_000081:output mode blocked by local_context_required"]

```

Current statuses are:

- `parse_success`
- `precondition_blocked`
- `unprobed`
- `help_text`
- `unvalidated`
- `parse_failed`

## Preconditions

Current precondition labels are:

| Precondition | Meaning |
|---|---|
| `auth_required` | The target appears to require login, credentials, tokens, or an authenticated profile. |
| `local_context_required` | The target appears to require a repository, workspace, project directory, configured profile, or similar current-directory context. |
| `fixture_required` | The target appears to require safe operands, required flags, resource identifiers, field values, or fixture data before behavior or output can be validated. |
| `network_unavailable` | The target appears to need a network service that is unavailable or constrained. |
| `runtime_dependency_unavailable` | The target appears to need a local daemon, socket, service, plugin, executable, or runtime dependency. |

Preconditions are inferred from target diagnostics and process status. They are not direct runtime network traces or subprocess traces.

## Artifact Map

Files generated by `cliare describe <artifact-dir> --write`:

```text
artifact-map.json
artifact-map.md
```

`artifact-map.json` has schema `cliare.artifact-map.v1` and describes an artifact directory. It is a navigation aid, not a score input.

Current top-level fields include:

```json
{
  "schema_version": "cliare.artifact-map.v1",
  "cliare_version": "0.1.4",
  "generated_at": "2026-06-15T21:11:30Z",
  "folder": ".cliare/cliare-on-cliare",
  "artifact_kind": "measurement",
  "health": {
    "status": "ok",
    "warnings": []
  },
  "navigation": [],
  "summaries": {},
  "missing_required": [],
  "files": [],
  "written_files": []
}
```

The artifact map should be safe to generate for partial folders. Missing required artifacts are reported through `missing_required` and health warnings.

## Runtime Context Suite

Context measurements store each context as a distinct measurement artifact directory:

```text
<suite-root>/contexts/<context-name>/
```

`runtime-context.json` records the context for each measurement. `cliare context compare <context-dir>... --write` writes:

```text
context-suite.json
context-compare.md
```

The context-suite artifact compares scores, preconditions, and artifact directories across contexts. It does not flatten persona files into one namespace.

## Schema Versions

Current schema names used by these artifacts include:

```text
cliare.evidence.v1
cliare.command-shape.v1
cliare.command-index.v1
cliare.scorecard.v1
cliare.score-model.v1
cliare.artifact-map.v1
cliare.runtime-context.v1
cliare.context-suite.v1
cliare.issue-list.v2
```

Schema versions are separate from score model versions. Current command-shape inference identifies itself as `cliare-generic-claims-v0`; current scoring uses the bundled `cliare-score-v0` model.

## Current Non-Goals

The current implementation does not emit:

- separate network event records
- separate classifier annotation records
- separate redaction records
- separate stdout/stderr artifact files
- tool-export artifacts
- JSON Schema value contracts for every positional or flag
- subprocess event records
- contradiction objects as a first-class shape section

If any of those become implemented artifacts, update the source code and generated examples first, then update this document from real output.
