# 20 - Persona Outcome Packets

> **Scope:** Current persona-specific reports generated from CLIARE measurement artifacts.
> **Status:** Current implementation reference with future extensions called out explicitly.

---

## Summary

CLIARE turns one measurement run into role-specific action packets.

A scorecard tells a team how a CLI measured. Persona packets tell a specific reviewer what to do next:

- maintainers get an agent-readiness fix queue
- harness builders get command-routing and exposure guidance
- platform teams get CI and policy posture
- security reviewers get side-effect and precondition evidence
- OSS maintainers get publishability guidance
- DevRel gets claims that are safe to make
- researchers get labeling and calibration-readiness context

Runtime probing remains separate from persona reporting. `cliare measure` runs the target binary and writes the measured artifacts. Persona reports are deterministic projections over those artifacts; they do not rerun the target CLI.

## Current Artifact Flow

A normal measurement writes the core artifacts and all persona packets:

```text
.cliare/<target>/
  evidence.jsonl
  shape.json
  command-index.json
  command-index.md
  scorecard.json
  report.md
  summary.md
  findings.sarif
  junit.xml
  runtime-context.json
  issues.json
  issues.md
  persona-maintainer.json
  persona-maintainer.md
  persona-harness.json
  persona-harness.md
  persona-platform.json
  persona-platform.md
  persona-security.json
  persona-security.md
  persona-oss.json
  persona-oss.md
  persona-devrel.json
  persona-devrel.md
  persona-research.json
  persona-research.md
```

`artifact-map.json`, `artifact-map.md`, `README.md`, and `AGENT_SKILL.md` are guide artifacts generated by `cliare describe --write` or refreshed by `cliare report --write`. They are useful navigation artifacts, but they are not score inputs.

The core loop is:

```sh
cliare measure ./mycli --out .cliare/mycli --profile standard
cliare report maintainer --out .cliare/mycli
cliare report harness --out .cliare/mycli --format json
cliare report security --out .cliare/mycli --write
```

## Runtime Context Suites

When a target is measured across contexts, reports are scoped to one concrete measurement context.

Example layout:

```text
.cliare/mycli/
  context-suite.json
  context-compare.md
  contexts/
    clean/
      scorecard.json
      command-index.json
      persona-maintainer.md
    workspace/
      scorecard.json
      command-index.json
      persona-maintainer.md
```

These commands address the same context bundle:

```sh
cliare report maintainer --out .cliare/mycli --context workspace
cliare report maintainer --out .cliare/mycli/contexts/workspace
```

If `--out` points at a context-suite root and more than one persisted context exists, CLIARE requires `--context` instead of guessing.

## Personas

The current persona enum is closed and product-facing:

```rust
pub enum Persona {
    Maintainer,
    Harness,
    Platform,
    Security,
    Oss,

    Research,

```

The CLI exposes the same values:

```sh
cliare report maintainer --out .cliare/mycli
cliare report harness --out .cliare/mycli
cliare report platform --out .cliare/mycli
cliare report security --out .cliare/mycli
cliare report oss --out .cliare/mycli
cliare report devrel --out .cliare/mycli
cliare report research --out .cliare/mycli
```

| Persona | Primary Question |
| --- | --- |
| `maintainer` | What should change in the CLI to improve agent readiness? |
| `harness` | Which command subset is ready to expose to agents? |
| `platform` | Can this CLI pass an internal automation quality gate? |
| `security` | What runtime evidence matters for approval or restriction? |
| `oss` | Is this scorecard ready to publish credibly? |
| `devrel` | Which public readiness claims are supported by evidence? |
| `research` | Can this run support future replay, labeling, and calibration? |

## Report Command

`cliare report` reads an existing measurement artifact directory.

Current options:

```sh
cliare report <persona> --out .cliare/mycli
cliare report <persona> --out .cliare/mycli --format markdown
cliare report <persona> --out .cliare/mycli --format json
cliare report <persona> --out .cliare/mycli --format bundle
cliare report <persona> --out .cliare/mycli --write
```

Focused views:

```sh
cliare report maintainer --out .cliare/mycli --area output-contracts
cliare report maintainer --out .cliare/mycli --issue issue.output_mode_unprobed
cliare report security --out .cliare/mycli --area safety --with-evidence
```

Supported areas:

```text
output-contracts
preconditions
command-discovery
help-coverage
compatibility
diagnostics
execution
safety
coverage
policy
publishing
calibration
```

Output behavior:

- default format is Markdown
- `--format json` prints the packet or focused drill-down as JSON
- `--format bundle` prints Markdown plus embedded JSON
- `--write` writes `persona-<persona>.md`, `persona-<persona>.json`, `issues.md`, and `issues.json`
- `--write` also refreshes measurement guide artifacts through the artifact-guide writer
- without `--write`, the selected representation prints to stdout only

## Packet Schema

Persona packet JSON is the automation contract:

```text
schema_version: cliare.persona-outcome.v1
```

Current top-level fields:

```text
schema_version
persona
persona_title
primary_question
target
source_artifacts
summary
run_recommendations
top_issues
reviewed_issues
action_items
command_health
score
coverage
evidence_summary
notes
```

`source_artifacts` points at:

- artifact directory
- `evidence.jsonl`
- `shape.json`
- `command-index.json`
- `command-index.md`
- `scorecard.json`

`summary` includes:

- score, score model, and score status
- measured and max score weight
- finding and shape-gap counts
- command discovery, confirmation, and precondition-blocked counts
- command readiness buckets: ready, conditional, needs fixture, blocked, candidate
- output contract and parse-success counts
- side-effect and credential-like side-effect counts
- precondition-blocked probe counts
- traversal profile fields, max depth, max probes, completed probes, frontier, stop reason, and completion state

`score` and `coverage` are copied from measured artifacts. Persona packets do not recompute the score independently.

## Issues And Dispositions

Persona packets are built from the same issue ledger:

```text
issues.json
issues.md
```

The packet splits issues into:

- `top_issues`: action-required issues relevant to the persona
- `reviewed_issues`: issues muted or reviewed through maintainer dispositions
- `action_items`: flattened, persona-prioritized work items

Dispositions do not delete findings from evidence. They change whether a finding remains action-required for persona review.

This matters for maintainers: a team can mark an item as accepted, rejected, deferred, or fixture-required and then rerun `cliare report` to see the persona queue reflect that decision without losing evidence.

## Command Health

`command_health` is the command-level bridge between the scorecard and remediation.

Each entry is derived from the command index and includes command readiness details such as:

- command path and argv form
- summary
- runtime state
- confidence
- suitability bucket
- suitability reasons
- parameters
- preconditions
- output contracts
- gaps
- evidence references

Maintainers use this to identify weak commands. Harness builders use it to decide which command subset is safe enough to expose to agents.

## Persona Semantics

### Maintainer

Focus:

- output contracts
- command discovery
- grammar
- recovery
- safety findings that affect agent readiness

Typical questions:

- Which commands need better help?
- Which advertised output modes need fixtures or parsing fixes?
- Which commands are blocked by runtime preconditions?
- Which side effects should be made read-only or documented?

### Harness

Focus:

- safe command exposure
- parseable output
- command confidence
- preconditions
- side effects

Typical questions:

- Which commands can an agent route to now?
- Which commands require auth, local context, or fixtures?
- Which commands should be excluded?
- Which output contracts are machine-readable and validated?

### Platform

Focus:

- CI gate posture
- policy readiness
- traversal completion
- release-blocking issues

Typical questions:

- Should this run pass an internal automation gate?
- Which dimensions or findings should block release?
- Which guard command should be added to CI?

### Security

Focus:

- filesystem side effects
- credential-like writes
- auth and local-context preconditions
- incomplete traversal under review

Typical questions:

- What did safe probes write?
- Did any path look credential-related?
- Is absence of evidence being mistaken for absence of risk?
- Which command paths should be restricted from agent exposure?

### OSS

Focus:

- public scorecard credibility
- README/release-note language
- contributor-friendly improvements
- provisional-run warnings

Typical questions:

- Is this scorecard publishable?
- What can the project safely claim?
- What should be fixed before public readiness messaging?

### DevRel

Focus:

- evidence-backed public story
- claims to avoid
- release narrative
- calibration and leaderboard caveats

Typical questions:

- What public claims are supported by evidence?
- What claims would overstate the current run?
- Which improvements would create the strongest readiness story?

### Research

Focus:

- labeling readiness
- model and evidence provenance
- budget and traversal limits
- calibration gaps

Typical questions:

- Which parts of the run are useful for truth-set labeling?
- Which model versions and budgets produced the artifacts?
- What uncertainty should be cited?
- What future replay/rescore work would make this easier?

## Run Recommendations

Persona packets include evidence-driven `run_recommendations`.

Examples:

```sh
cliare measure ./mycli --out .cliare/mycli --profile deep --max-depth 8 --max-probes 1000 --concurrency 8 --refresh
cliare guard ./mycli --baseline .cliare/baseline.scorecard.json --policy cliare.policy.json --out .cliare/mycli
cliare report security --out .cliare/mycli --write
cliare report harness --out .cliare/mycli --format json --write
```

Recommendations are generated from observed state:

| Evidence | Recommendation |
| --- | --- |
| traversal incomplete | Run a deeper measurement. |
| observed max depth reached max depth | Increase `--max-depth`. |
| budget exhausted | Increase `--max-probes`. |
| no machine-readable output contracts | Re-measure after adding JSON/YAML output modes. |
| side effects or auth gates under review | Generate the security packet. |
| platform persona | Add a guard command with baseline and policy. |
| harness persona | Persist machine-readable harness packet. |
| research persona | Produce a deeper evidence set for labeling. |

## Depth And Coverage

Persona packets expose traversal pressure because incomplete traversal is uncertainty, not failure by itself.

Relevant fields include:

- traversal profile
- max depth
- observed max depth
- max probes
- probes scheduled and completed
- frontier remaining
- highest pending expected value
- candidates skipped by depth
- candidates skipped by convergence
- probes skipped by budget
- traversal stop reason
- traversal complete

Useful run patterns:

| Scenario | Command Pattern |
| --- | --- |
| Pull request smoke check | `cliare measure ./mycli --profile quick --out .cliare/mycli` |
| Release readiness | `cliare measure ./mycli --profile standard --out .cliare/mycli` |
| Large public CLI | `cliare measure ./mycli --profile deep --max-depth 8 --max-probes 1000 --out .cliare/mycli` |
| Detached long run | `cliare measure ./mycli --profile deep --out .cliare/mycli --detach` |
| Job inspection | `cliare jobs status --out .cliare/mycli` |

## Current Quality Bar

Persona reporting is part of the public CLIARE artifact contract. It must remain:

- deterministic for the same input artifacts
- schema-versioned
- explicit about unknown and incomplete states
- grounded in scorecard, shape, command index, issue ledger, and evidence
- separate from probing
- free of source-code or framework assumptions
- conservative about safety and public claims
- compatible with maintainer dispositions

Future extensions may convert harness packets into tool definitions, adapter scaffolds, or installable agent skills. Those conversions should remain downstream of the persona packet and should not change the measured evidence.
